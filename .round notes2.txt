While your instinct to look at the window.currentCourse data bridge makes perfect sense, I have to gently correct that theory. Your Object.defineProperty getter and the spread operator (...c) are working exactly as intended. We know window.currentCourse isn't missing its holes array because the very first line of loadHole evaluates course.holes.length. If holes were stripped, the engine would crash right there, long before it ever tried to look for .zones.
The actual culprit is a silent failure caused by a missing property in "The Pasture" data, which poisons a global variable and detonates inside your targeting function.
The Trace: How the Crash Happens
1. 
The Missing Property: In data_ag.js, the hole objects for "The Holodeck Links" and "Pebble Beach" explicitly define a number property (e.g., number: 1). The newly generated objects for "The Pasture" do not.
2. 
The Poisoned Variable: When you click "Start Round!", the engine calls loadHole(1). Around line 360 in main_ag.js, you assign the global hole variable:

Copy code
hole = holeData.number; 
Because "The Pasture" lacks this property, the global hole variable silently becomes undefined.
3. 
The Detonation: A few lines later, loadHole calls window.updateTargetZone(). Inside that function, the engine attempts to pull the active hole data again:

Copy code
const holeData = window.currentCourse.holes[hole - 1];
Since hole is undefined, hole - 1 evaluates to NaN. Looking up holes[NaN] evaluates to undefined.
4. 
The Crash: The very next line checks if (holeData.zones). Trying to read .zones on undefined throws your exact fatal error.
 
The Fix
You don't need to touch zones, and you don't even need to go back and manually type number: into all 18 objects in data_ag.js. We can just make the engine self-healing so it never relies solely on the data file for the hole number.
Update your loadHole function in main_ag.js to fallback to the passed holeNumber argument:
